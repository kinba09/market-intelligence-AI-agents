How this platform works in production

Source connectors feed a normalized document stream

Pull data from RSS/news APIs, company websites, job boards, social APIs, and report uploads. Every record is transformed to a canonical schema, deduplicated, and versioned.

  • Batch: daily/hourly pulls for reports and site crawls
  • Near real time: webhook + streaming for high-value feeds
  • Store raw HTML/PDF/JSON in object storage for reprocessing

AI enrichment converts text into structured intelligence

NLP pipelines detect companies, products, people, events, sentiment, and impact. Entity linking maps mentions to canonical company IDs to avoid duplicates.

  • NER + relation extraction for who-did-what-when
  • Event taxonomy: funding, launch, pricing, hiring, partnerships
  • Importance scoring prioritizes analyst attention

Q&A uses grounded retrieval with strict citation controls

Query understanding builds retrieval intents, then combines BM25 and vector search. Results are reranked and passed to a constrained synthesis prompt.

  • Metadata filters: date, source quality, company, geography
  • Cross-encoder rerank improves precision@k
  • Every claim tied to source chunk IDs and links

Agents monitor deltas and trigger scored alerts

Agents compare latest signals against baseline behavior. A rule + ML score gates whether to send an alert, generate a brief, or wait for more evidence.

  • Detect pricing page changes with semantic diffs
  • Hiring spike alerts from role velocity anomalies
  • Daily/weekly digests with confidence and evidence

1. System Architecture

End-to-end architecture and data flow

Built for a solo builder: modular, practical, and production-minded.

  - Entity linking + importance scoring
  v

  - OpenSearch (BM25 + filters)
  - Qdrant (vector index)
  - Postgres (entities/events/relations)

  - Hybrid retrieval + rerank

  v

  - Monitoring agents
  - Alert policy engine
  - Report synthesis agents
  v
[Output Layer]
  - Web dashboard
  - Slack/email alerts
  - Scheduled competitor briefs

Data flow (request + background)

A) Background Intelligence Loop
Ingest -> Normalize -> Enrich -> Index -> Score -> Alert candidates

B) User Q&A Loop
Question -> Query Understanding -> Hybrid Retrieve -> Rerank
-> Grounded Prompt -> Answer + Citations

C) Monitoring Loop
Watchlist Company -> Delta Detection -> Confidence Score
-> Trigger Policy -> Alert or Hold -> Daily Summary

Key design choice: separate raw artifact storage from derived intelligence so you can re-run extraction models when prompts or taxonomies improve.

3. RAG Design

Grounded answers with hybrid retrieval

Chunking strategy

  • Primary chunk size: 500-800 tokens with 80-token overlap.
  • Semantic boundaries: split by heading/section before token slicing.
  • Metadata-aware chunks include `source_type`, `company_id`, `published_at`, `language`, `confidence`.
  • Special handling for tables and pricing pages: preserve row/field context.

Why this works

Semantic-first chunking reduces context fragmentation, while metadata enables high-precision filtering for analyst-grade responses.

Hybrid retrieval formula

final_score = 0.45 * bm25_norm + 0.45 * vector_cosine + 0.10 * freshness_boost
  • Retrieve top 40 BM25 + top 40 vector results.
  • Deduplicate by `chunk_id`, then merge scores.
  • Rerank top 40 with a cross-encoder, keep top 8-12 context chunks.

Prompt template (grounded)

SYSTEM: You are a market intelligence analyst.
Use only provided evidence. If evidence is insufficient, say so.
Return JSON: {answer, confidence, citations[]}.

USER QUESTION: {question}
EVIDENCE:
{top_chunks_with_source_ids}

RULES:
- Every major claim needs at least one citation ID.
- No external knowledge.
- Prefer recent evidence when conflicts exist.

4. Agent Design

Multi-agent workflow with strict handoffs

Agent graph

[Query Understanding Agent]
  input: user question
  output: intent, entities, filters, answer_type
        |
        v
[Retrieval Agent]
  tools: search_bm25, search_vector, fetch_document
  output: candidate_chunks + ranking_features
        |
        v
[Validation Agent]
  tasks: contradiction check, source diversity, evidence sufficiency
  output: validated_context or follow-up question
        |
        v
[Synthesis Agent]
  tasks: produce final answer, citations, confidence, action items
  output: analyst-ready response JSON

Tool-calling contract

  • All tool inputs/outputs are Pydantic models for reliability.
  • Agent state persisted as JSON in Redis/Postgres for replay/debugging.
  • Validation agent can reject synthesis if citations are weak.
  • Fallback mode returns "insufficient evidence" instead of guessing.

5. Data Pipeline

Raw content to structured intelligence

Pipeline stages

  1. Ingest raw artifact + metadata capture.
  2. Normalize text (boilerplate removal, language detect).
  3. Deduplicate by URL fingerprint + semantic near-duplicate check.
  4. Entity extraction (company, product, role, region, person).
  5. Event detection (funding, launch, pricing, hiring, partnership).
  6. Sentiment + stance classification (positive/neutral/negative).
  7. Importance scoring and watchlist matching.
  8. Write document/chunks/events to serving stores.

Importance scoring heuristic

importance =
  0.35 * source_credibility
+ 0.25 * event_severity
+ 0.20 * watchlist_match
+ 0.10 * social_velocity
+ 0.10 * novelty

Use this heuristic initially, then replace with a learned ranking model once you have analyst feedback labels.

Event schema (example)

6. Alert System

Real-time competitive signal detection

Trigger families

Competitor changes

Pricing page diff, product page changelog, docs updates, roadmap shifts.

Funding signals

Press release + SEC filing + investor post triangulation for confidence.

Hiring spikes

Role velocity anomalies by team/location over 14-day baseline.

Alert policy logic

if importance >= 0.75 and confidence >= 0.70 and sources >= 2:
    send_alert(channel="slack", priority="high")
elif importance >= 0.60 and confidence >= 0.55:
    queue_digest(priority="medium")
else:
    hold_for_more_evidence()

Add cooldown windows to suppress duplicate alerts for the same company + event type within a configurable period (e.g., 6 hours).

7. Database Design

Schemas and indexing strategy

Core tables

companies(
  id PK, name, domain, industry, headquarters,
  watchlist_tier, created_at, updated_at
)

documents(
  id PK, source_url, source_type, title,
  raw_object_key, published_at, language,
  hash, ingestion_time
)

chunks(
  id PK, document_id FK, chunk_index,
  text, token_count, embedding_id,
  company_ids[], event_tags[], created_at
)

events(
  id PK, company_id FK, event_type,
  event_time, sentiment, importance,
  summary, evidence_chunk_ids[], created_at
)

Indexing plan

  • `documents(source_url)` unique index for dedupe.
  • `documents(published_at DESC)` for recency retrieval.
  • `events(company_id, event_time DESC)` for timelines.
  • `events(event_type, importance DESC)` for alerts.
  • OpenSearch index on `chunks.text` with BM25 + metadata facets.
  • Qdrant payload filters on company IDs, date, source type.

Keep high-volume text in OpenSearch/Qdrant and use Postgres for relational truth.

8.1 Ingestion pipeline (FastAPI + async fetch)

from fastapi import FastAPI
from pydantic import BaseModel
import httpx
from datetime import datetime

app = FastAPI()

class IngestRequest(BaseModel):
    url: str
    source_type: str

async def fetch_html(url: str) -> str:
    async with httpx.AsyncClient(timeout=20.0) as client:
        r = await client.get(url)
        r.raise_for_status()
        return r.text

@app.post("/ingest")
async def ingest(req: IngestRequest):
    html = await fetch_html(req.url)
    # TODO: clean_text, extract metadata, dedupe hash
    doc = {
        "source_url": req.url,
        "source_type": req.source_type,
        "raw_html": html,
        "ingested_at": datetime.utcnow().isoformat()
    }
    # TODO: write raw artifact to S3 + metadata to Postgres
    return {"status": "queued", "document": doc}

8.2 Embeddings + Qdrant upsert

from sentence_transformers import SentenceTransformer
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct

model = SentenceTransformer("BAAI/bge-large-en-v1.5")
qdrant = QdrantClient(url="http://localhost:6333")


def upsert_chunks(chunks):
    texts = [c["text"] for c in chunks]
    vectors = model.encode(texts, normalize_embeddings=True)

    points = []
    for c, v in zip(chunks, vectors):
        points.append(
            PointStruct(
                id=c["chunk_id"],
                vector=v.tolist(),
                payload={
                    "document_id": c["document_id"],
                    "company_ids": c.get("company_ids", []),
                    "published_at": c.get("published_at"),
                    "source_type": c.get("source_type")
                }
            )
        )

    qdrant.upsert(collection_name="market_chunks", points=points)

8.3 Hybrid retrieval + simple RAG

def hybrid_retrieve(question, bm25_client, qdrant_client, embed_fn, k=40):
    bm25_hits = bm25_client.search(question, top_k=k)
    vec = embed_fn(question)
    vec_hits = qdrant_client.search(
        collection_name="market_chunks",
        query_vector=vec,
        limit=k
    )

    # score fusion
    scores = {}
    for h in bm25_hits:
        scores[h.id] = scores.get(h.id, 0) + 0.45 * h.norm_score
    for h in vec_hits:
        scores[h.id] = scores.get(h.id, 0) + 0.45 * h.score

    ranked_ids = sorted(scores, key=scores.get, reverse=True)[:40]
    return ranked_ids


def answer_with_citations(question, retriever, llm, chunk_store):
    ids = retriever(question)
    context = [chunk_store[i] for i in ids[:10]]
    prompt = f"""
Use only evidence below. Attach citation IDs for every key claim.
Question: {question}
Evidence:\n{context}
Output JSON: {{"answer": str, "citations": list[str], "confidence": float}}
"""
    return llm.invoke(prompt)

8.4 Basic agent loop (LangGraph-style pseudocode)

from pydantic import BaseModel

class AgentState(BaseModel):
    question: str
    intent: dict | None = None
    retrieved: list | None = None
    validated: list | None = None
    answer: dict | None = None


def query_understanding(state: AgentState):
    state.intent = {"entities": [], "time_window": "90d", "type": "analysis"}
    return state


def retrieve(state: AgentState):
    state.retrieved = hybrid_retrieve(state.question, bm25, qdrant, embed)
    return state


def validate(state: AgentState):
    state.validated = state.retrieved[:10]  # add contradiction/evidence checks
    return state


def synthesize(state: AgentState):
    state.answer = answer_with_citations(state.question, lambda q: state.validated, llm, chunks)
    return state

9. Evaluation

How to measure quality before scaling

Retrieval metrics

  • nDCG@10 for ranking quality.
  • MRR for first-relevant-chunk position.
  • Recall@20 against labeled evidence sets.
  • Latency p50/p95 for retrieval and full answer pipeline.

Protocol

Build a benchmark of ~150 real analyst questions with gold citations. Run nightly and compare against previous prompt/model settings.

Answer quality metrics

  • Faithfulness: percent of claims supported by cited chunks.
  • Groundedness: no unsupported statements.
  • Citation precision: cited chunks directly justify the claim.
  • Analyst utility rating (1-5) for report usefulness.

Practical test harness

pytest tests/evals/test_retrieval.py
pytest tests/evals/test_grounded_answers.py
python scripts/run_eval_dashboard.py

10. MVP Plan

Build order for 1-2 weeks, then scale

Week 1-2 (high-impact MVP)

  1. Ingest 3 sources: news API, company websites, job postings.
  2. Store raw docs + metadata in Postgres and object storage.
  3. Chunk + embed + index in OpenSearch and Qdrant.
  4. Ship Q&A API with citations and confidence score.
  5. Launch dashboard with company watchlist + recent events.
  6. Add one alert type: pricing page change.

Goal: prove end-to-end loop works reliably and is demo-ready for recruiters.

Scaling roadmap (later)

  1. Add more connectors (social, earnings calls, app reviews).
  2. Train event ranking using analyst feedback.
  3. Expand agents for auto-generated competitor briefs.
  4. Add multi-tenant auth, billing, and usage controls.
  5. Implement robust monitoring, caching, and cost optimization.
  6. Introduce human-in-the-loop review queues for critical alerts.

This progression keeps complexity proportional to traction while preserving a production path.

Portfolio positioning

Present this as a practical AI systems project: deterministic data engineering + LLM reasoning + measurable retrieval quality. Recruiters care most about grounded outputs, clear architecture, and evaluation discipline.
